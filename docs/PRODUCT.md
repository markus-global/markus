# Markus 产品愿景

> "我们不是在造工具，而是在雇佣同事。"

---

## 第一性原理

### 核心问题：现有 AI 助手都是工具，不是同事

| 维度 | 工具（现状） | 同事（Markus 目标） |
|------|-------------|-------------------|
| 存在方式 | 用时打开，不用关闭 | 始终在线，像远程员工 |
| 交互方式 | 点击按钮、填写表单 | 对话指令、异步协作 |
| 主动性 | 被动等待人类指令 | 主动发现问题、推进工作 |
| 记忆 | 每次对话从零开始 | 记住所有项目上下文 |
| 协作 | 人→机 单向 | 人⇄机、机⇄机 多向 |
| 失败处理 | 报错给人 | 求助同事或发布悬赏 |

### 数字员工的七个特征

1. **入职（Onboard）** — 给 TA 一份工作描述、一个工位、认识团队
2. **自主工作** — 不需要"启动"才干活，早上自动检查邮件、看 Issue
3. **主动沟通** — 遇到问题主动在群里说，完成任务主动汇报
4. **使用工具** — 打开浏览器看文档、登 GitHub 提 PR、用飞书发消息
5. **有记忆** — 记得上周讨论了什么、项目的技术架构、同事的偏好
6. **会学习** — 第一次做某件事需要指导，做过的事情以后更高效
7. **知道边界** — 做不了的事情会说"我搞不定，需要人帮忙"

## 目标用户

| 用户类型 | 核心痛点 | Markus 解法 |
|---------|---------|------------|
| 一人公司 / 自由职业者 | 一个人要做开发、设计、营销、客服、财务 | AI 团队：CTO + 设计师 + 营销 + 客服 |
| 中小公司（5-50 人） | 技术人才贵且难招，重复工作多 | AI 员工填补角色空缺，自动化日常 |
| 大公司（50+） | 人力成本高，跨部门协调慢 | AI 员工做重复性工作，人类专注创造性 |

## 核心体验原则

### 组织结构

- 每个 AI 组织有一个 **组织负责人（Manager Agent）**，负责路由、协调、汇报
- 普通员工（Worker Agent）负责具体领域任务
- 人类用户有明确角色（owner / admin / member）和对应权限

### 任务是工作的单元

- Agent 的所有有意义的工作必须绑定到 Task
- Task 有完整生命周期：pending → in_progress → completed
- 支持子任务拆解和跨 Agent 协作
- 心跳复盘：Agent 定期回顾自己的 Task 列表

### 沟通是协作的基础

- 团队频道（#general / #dev）：人类和 Agent 共同参与
- Agent 间消息（A2A）：自动任务委派和进度同步
- 人类私信：人与人之间的协作，不打扰 Agent

## 两层架构定位

```
┌──────────────────────────────────────────────────────────┐
│                    Markus Cloud（未来）                    │
│   SaaS · 多租户 · 计费 · 企业级 · 全球部署               │
├──────────────────────────────────────────────────────────┤
│                 Markus OSS（当前）✅                       │
│   开源 · 自托管 · 单组织 · 简洁 · 可扩展                  │
└──────────────────────────────────────────────────────────┘
         Cloud 是 OSS 的超集，共享全部核心代码
```

OSS 设计原则：
- `docker compose up` 一键启动
- 单组织模型，不引入多租户复杂度
- 认证可选（内网信任环境可关闭）
- 数据完全归用户所有，可随时导出
