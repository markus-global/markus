# 代码评审清单

## 概述
本清单用于指导代码评审过程，确保评审全面且一致。评审者应在评审过程中逐项检查。

## 1. 功能实现检查

### 1.1 需求符合性
- [ ] 功能是否完整实现了需求？
- [ ] 是否有遗漏的功能点？
- [ ] 功能是否按预期工作？

### 1.2 边界情况处理
- [ ] 是否处理了空值/空数组？
- [ ] 是否处理了错误输入？
- [ ] 是否处理了异常情况？
- [ ] 是否有内存/性能边界考虑？

### 1.3 用户体验
- [ ] 用户界面是否友好？
- [ ] 错误提示是否清晰？
- [ ] 加载状态是否合理？
- [ ] 响应时间是否可接受？

## 2. 代码质量检查

### 2.1 代码结构
- [ ] 代码是否模块化？
- [ ] 函数/方法是否职责单一？
- [ ] 类设计是否合理？
- [ ] 文件组织是否清晰？

### 2.2 代码可读性
- [ ] 变量/函数命名是否清晰？
- [ ] 注释是否充分且有用？
- [ ] 代码逻辑是否容易理解？
- [ ] 是否有复杂的嵌套逻辑？

### 2.3 代码复用
- [ ] 是否有重复代码？
- [ ] 是否可以提取公共函数？
- [ ] 是否使用了合适的抽象？
- [ ] 是否有不必要的依赖？

## 3. 测试检查

### 3.1 测试覆盖
- [ ] 是否有单元测试？
- [ ] 测试是否覆盖主要场景？
- [ ] 是否有边界情况测试？
- [ ] 测试覆盖率是否达标？

### 3.2 测试质量
- [ ] 测试是否独立可运行？
- [ ] 测试是否清晰易读？
- [ ] 测试是否验证了正确行为？
- [ ] 是否有集成测试？

### 3.3 测试维护性
- [ ] 测试是否容易维护？
- [ ] 测试数据是否合理？
- [ ] 测试是否避免了实现细节依赖？
- [ ] 测试失败信息是否清晰？

## 4. 安全检查

### 4.1 输入验证
- [ ] 是否验证了所有用户输入？
- [ ] 是否防止了SQL注入？
- [ ] 是否防止了XSS攻击？
- [ ] 是否防止了CSRF攻击？

### 4.2 数据保护
- [ ] 敏感信息是否加密？
- [ ] 密码是否安全存储？
- [ ] API密钥是否妥善管理？
- [ ] 日志是否包含敏感信息？

### 4.3 权限控制
- [ ] 是否检查了用户权限？
- [ ] 是否有越权访问风险？
- [ ] 是否遵循最小权限原则？
- [ ] 是否有认证/授权漏洞？

## 5. 性能检查

### 5.1 响应时间
- [ ] API响应时间是否合理？
- [ ] 是否有慢查询？
- [ ] 是否有不必要的计算？
- [ ] 是否有缓存机制？

### 5.2 资源使用
- [ ] 内存使用是否合理？
- [ ] 是否有内存泄漏风险？
- [ ] 数据库连接是否有效管理？
- [ ] 文件句柄是否及时释放？

### 5.3 可扩展性
- [ ] 代码是否支持水平扩展？
- [ ] 是否有单点故障？
- [ ] 是否有并发问题？
- [ ] 是否考虑了负载均衡？

## 6. 维护性检查

### 6.1 文档
- [ ] 代码是否有适当的注释？
- [ ] API是否有文档？
- [ ] 配置是否有说明？
- [ ] 部署步骤是否文档化？

### 6.2 配置管理
- [ ] 配置是否与环境分离？
- [ ] 是否有默认配置？
- [ ] 配置验证是否完善？
- [ ] 敏感配置是否安全？

### 6.3 监控和日志
- [ ] 是否有适当的日志记录？
- [ ] 错误是否可追踪？
- [ ] 是否有监控指标？
- [ ] 告警配置是否合理？

## 7. 兼容性检查

### 7.1 向后兼容
- [ ] API变更是否向后兼容？
- [ ] 数据库迁移是否安全？
- [ ] 配置变更是否有迁移路径？
- [ ] 是否影响了现有功能？

### 7.2 环境兼容
- [ ] 是否支持目标Node.js版本？
- [ ] 是否支持目标浏览器？
- [ ] 是否支持目标操作系统？
- [ ] 依赖版本是否兼容？

## 8. 代码规范检查

### 8.1 代码风格
- [ ] 是否符合项目代码风格？
- [ ] 缩进和空格是否正确？
- [ ] 行长度是否合理？
- [ ] 导入/导出是否规范？

### 8.2 TypeScript规范
- [ ] 类型定义是否完整？
- [ ] 是否有any类型滥用？
- [ ] 泛型使用是否合理？
- [ ] 类型断言是否必要？

### 8.3 错误处理
- [ ] 错误是否被适当捕获？
- [ ] 错误信息是否清晰？
- [ ] 是否有全局错误处理？
- [ ] 是否避免了静默失败？

### 9.1 评审结论
- [ ] **通过**：所有检查项通过，可以合并
- [ ] **需要修改**：有需要修复的问题
- [ ] **拒绝**：存在严重问题，需要重新设计

## 使用说明

### 评审流程
1. **初步评审**：快速浏览代码，了解整体结构
2. **详细评审**：使用本清单逐项检查
3. **记录反馈**：在评审工具中记录发现的问题
4. **总结评估**：给出评审结论和建议

### 评审重点
根据代码变更类型，重点关注不同方面：

**新功能开发**
- 功能完整性
- 测试覆盖
- 用户体验

**Bug修复**
- 根本原因分析
- 修复方案正确性
- 回归测试

**重构优化**
- 性能提升
- 代码质量改进
- 向后兼容性

**依赖更新**
- 版本兼容性
- 安全影响
- 功能影响

### 评审工具
- GitHub/GitLab代码评审功能
- 行内评论和建议
- 自动化检查结果
- 测试覆盖率报告
