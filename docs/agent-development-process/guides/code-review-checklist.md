# 代码评审检查清单

## 概述
本检查清单用于指导Agent代码评审过程，确保代码质量和一致性。

## 评审前准备

### 1. 环境准备
- [ ] 已拉取最新代码到本地
- [ ] 已安装项目依赖
- [ ] 能够运行测试和构建

### 2. 理解上下文
- [ ] 阅读任务描述和需求
- [ ] 查看相关文档和设计
- [ ] 了解代码变更的背景

## 代码质量检查

### 1. 代码规范
- [ ] **命名规范**
  - [ ] 变量、函数、类名是否清晰有意义
  - [ ] 命名是否符合项目约定（camelCase、PascalCase等）
  - [ ] 命名是否一致（相同概念使用相同术语）

- [ ] **代码结构**
  - [ ] 函数和方法是否职责单一
  - [ ] 代码模块化是否合理
  - [ ] 是否有重复代码可以提取
  - [ ] 文件组织是否清晰

- [ ] **代码复杂度**
  - [ ] 函数长度是否合理（建议不超过50行）
  - [ ] 嵌套深度是否合理（建议不超过3层）
  - [ ] 圈复杂度是否可接受

### 2. 类型安全（TypeScript）
- [ ] **类型定义**
  - [ ] 是否正确定义了类型
  - [ ] 是否使用了合适的类型（interface vs type）
  - [ ] 泛型使用是否恰当

- [ ] **类型检查**
  - [ ] 是否有any类型（应尽量避免）
  - [ ] 类型断言是否必要和安全
  - [ ] 空值处理是否完善

### 3. 错误处理
- [ ] **异常处理**
  - [ ] 是否处理了可能的异常
  - [ ] 错误消息是否清晰有用
  - [ ] 是否记录了必要的日志

- [ ] **边界条件**
  - [ ] 是否处理了空值、空数组等边界情况
  - [ ] 是否处理了无效输入
  - [ ] 是否考虑了并发和竞态条件

## 功能实现检查

### 1. 需求满足
- [ ] **功能完整性**
  - [ ] 是否实现了所有需求功能
  - [ ] 是否有遗漏的功能点
  - [ ] 功能实现是否正确

- [ ] **用户体验**
  - [ ] API设计是否合理
  - [ ] 错误处理是否友好
  - [ ] 性能是否可接受

### 2. 设计模式
- [ ] **架构设计**
  - [ ] 是否遵循了项目架构模式
  - [ ] 设计是否可扩展和可维护
  - [ ] 是否有更好的设计选择

- [ ] **代码复用**
  - [ ] 是否复用了现有代码
  - [ ] 是否创建了不必要的重复
  - [ ] 抽象层次是否合理

## 测试检查

### 1. 测试覆盖
- [ ] **测试完整性**
  - [ ] 是否为新功能添加了测试
  - [ ] 测试是否覆盖主要场景
  - [ ] 是否覆盖了边界条件

- [ ] **测试质量**
  - [ ] 测试是否清晰易懂
  - [ ] 测试是否独立（不依赖其他测试）
  - [ ] 测试命名是否清晰

### 2. 测试类型
- [ ] **单元测试**
  - [ ] 是否测试了核心逻辑
  - [ ] 是否使用了恰当的测试工具
  - [ ] Mock使用是否合理

- [ ] **集成测试**
  - [ ] 是否测试了组件集成
  - [ ] 是否测试了外部依赖
  - [ ] 测试环境配置是否正确

## 文档和注释

### 1. 代码注释
- [ ] **注释质量**
  - [ ] 复杂逻辑是否有注释说明
  - [ ] 注释是否准确和最新
  - [ ] 是否有TODO或FIXME注释需要处理

- [ ] **API文档**
  - [ ] 公共API是否有文档
  - [ ] 文档是否清晰完整
  - [ ] 是否有使用示例

### 2. 项目文档
- [ ] **README更新**
  - [ ] 是否更新了相关README
  - [ ] 安装和使用说明是否准确
  - [ ] 是否有变更记录

- [ ] **设计文档**
  - [ ] 是否有设计决策记录
  - [ ] 架构图是否更新
  - [ ] 接口文档是否完整

## 安全性和性能

### 1. 安全性
- [ ] **输入验证**
  - [ ] 是否验证了所有用户输入
  - [ ] 是否有SQL注入风险
  - [ ] 是否有XSS攻击风险

- [ ] **敏感信息**
  - [ ] 是否有硬编码的密钥或密码
  - [ ] 敏感信息是否安全存储
  - [ ] 日志中是否包含敏感信息

### 2. 性能
- [ ] **资源使用**
  - [ ] 是否有内存泄漏风险
  - [ ] 是否有性能瓶颈
  - [ ] 数据库查询是否优化

- [ ] **响应时间**
  - [ ] API响应时间是否可接受
  - [ ] 是否有不必要的阻塞操作
  - [ ] 缓存使用是否合理

## 工具和自动化

### 1. 代码质量工具
- [ ] **检查结果**
  - [ ] ESLint检查是否通过
  - [ ] TypeScript编译是否通过
  - [ ] 测试覆盖率是否达标

- [ ] **构建和部署**
  - [ ] 代码是否可以正常构建
  - [ ] 部署脚本是否需要更新
  - [ ] CI/CD流水线是否配置正确

## 评审流程

### 1. 评审沟通
- [ ] **评审意见**
  - [ ] 意见是否清晰具体
  - [ ] 是否提供了改进建议
  - [ ] 是否标记了优先级（必须修复、建议改进等）

- [ ] **讨论记录**
  - [ ] 重要讨论是否有记录
  - [ ] 决策是否有记录
  - [ ] 是否有待解决的问题

### 2. 评审结果
- [ ] **评审结论**
  - [ ] 是否批准合并
  - [ ] 是否需要修改后重新评审
  - [ ] 是否有阻塞性问题

## 评审最佳实践

### 1. 评审态度
- **建设性反馈**：提供具体的改进建议，而不是仅仅指出问题
- **尊重作者**：理解作者的意图和努力
- **学习机会**：把评审当作学习和分享的机会

### 2. 评审效率
- **小批量评审**：每次评审的代码量不宜过大（建议不超过400行）
- **及时评审**：在24小时内完成初步评审
- **专注评审**：避免中断，集中注意力

### 3. 评审工具
- **使用工具辅助**：利用代码质量工具发现常见问题
- **记录评审意见**：使用PR评论功能记录讨论
- **跟踪问题解决**：确保所有问题都得到解决

## 常见问题处理

### 1. 意见分歧
- **技术分歧**：基于事实和数据讨论，必要时寻求第三方意见
- **风格分歧**：遵循项目代码规范，保持一致性
- **优先级分歧**：明确问题的优先级和影响范围

### 2. 复杂变更
- **分阶段评审**：将复杂变更分解为多个PR
- **设计评审**：复杂功能先进行设计评审
- **原型验证**：复杂实现可以先提交原型验证

## 评审模板

### PR描述模板
```markdown
## 变更描述
[简要描述本次变更的目的和内容]

## 相关任务
- 任务ID: tsk_xxxxxxx
- 相关PR: #xxx

## 测试结果
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 代码质量检查通过
- [ ] 构建成功

## 变更类型
- [ ] Bug修复
- [ ] 新功能
- [ ] 重构
- [ ] 文档更新
- [ ] 其他

## 检查清单
- [ ] 代码符合项目规范
- [ ] 添加了必要的测试
- [ ] 更新了相关文档
- [ ] 考虑了向后兼容性
```

### 评审评论模板
```markdown
**问题描述**：[具体描述问题]
**影响**：[问题的影响范围]
**建议方案**：[具体的改进建议]
**优先级**：[高/中/低]
```

## 总结
代码评审是保证代码质量的重要环节。通过系统化的评审流程和检查清单，可以提高评审效率和质量，促进团队知识共享和代码质量提升。
