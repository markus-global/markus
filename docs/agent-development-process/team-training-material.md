## 详细培训内容

### 1. 开发流程概述

#### 1.1 为什么需要标准开发流程？
- **一致性**：确保所有团队成员遵循相同的工作方式
- **质量保证**：通过标准化检查提高代码质量
- **效率提升**：减少沟通成本和重复工作
- **知识传承**：新成员可以快速上手

#### 1.2 流程核心原则
1. **隔离开发**：每个任务在独立的worktree中开发
2. **质量优先**：代码质量检查贯穿整个开发过程
3. **协作评审**：代码必须经过同行评审才能合并
4. **持续改进**：定期回顾和优化流程

#### 1.3 流程阶段
```
任务获取 → 环境准备 → 代码开发 → 质量检查 → 提交评审 → 合并部署
```

### 2. Git Worktree深度实践

#### 2.1 Git Worktree概念
- **什么是Worktree**：同一个仓库的多个工作目录
- **优势**：
  - 隔离开发环境，避免冲突
  - 快速切换任务上下文
  - 保持主分支干净

#### 2.2 创建Worktree
```bash
# 使用工具创建
./tools/git-worktree/create-worktree.sh tsk_12345678 your-name

# 手动创建
git worktree add ../markus-tsk_12345678 main
cd ../markus-tsk_12345678
git checkout -b feature/tsk_12345678
```

#### 2.3 Worktree管理
```bash
# 列出所有worktree
git worktree list

# 切换到其他worktree
cd ../markus-other-task

# 清理完成的worktree
./tools/git-worktree/cleanup-worktree.sh worktree-name
```

#### 2.4 练习：创建和管理Worktree
**任务**：为任务`tsk_training_001`创建worktree
1. 使用工具创建worktree
2. 在worktree中创建功能分支
3. 进行一些代码修改
4. 提交修改
5. 列出所有worktree

### 3. 代码质量检查工具使用

#### 3.1 工具介绍
- **markus-check**：统一的代码质量检查工具
- **检查类型**：
  - Git Diff检查
  - TypeScript检查
  - ESLint检查
  - 测试检查
  - 安全检查

#### 3.2 基本使用
```bash
# 检查所有项目
npx @markus/quality-check check all

# 检查特定项目
npx @markus/quality-check check typescript
npx @markus/quality-check check eslint

# 查看帮助
npx @markus/quality-check --help
```

#### 3.3 配置检查
```json
// markus-check.config.json
{
  "gitDiff": {
    "enabled": true,
    "excludePatterns": ["*.md", "*.json"]
  },
  "typescript": {
    "enabled": true,
    "strict": true
  }
}
```

#### 3.4 练习：使用质量检查工具
**任务**：在之前创建的worktree中
1. 安装质量检查工具
2. 运行所有检查
3. 故意引入一些代码问题
4. 运行检查并查看结果
5. 修复发现的问题

### 4. 代码评审实践

#### 4.1 评审原则
- **建设性**：提供有帮助的反馈
- **尊重性**：尊重他人的工作和时间
- **全面性**：检查代码的各个方面
- **及时性**：在规定时间内完成评审

#### 4.2 评审清单使用
使用[代码评审清单](./code-review-checklist.md)确保评审全面：
1. **功能实现**：需求符合性、边界情况
2. **代码质量**：结构、可读性、复用性
3. **测试覆盖**：单元测试、集成测试
4. **安全检查**：输入验证、权限控制
5. **性能考虑**：响应时间、资源使用

#### 4.3 评审沟通技巧
- **明确问题**：具体指出问题所在
- **提供建议**：给出改进建议或示例
- **解释原因**：说明为什么需要修改
- **保持积极**：使用积极的语言

#### 4.4 练习：代码评审
**任务**：评审示例代码
1. 分成小组（2-3人一组）
2. 每组评审提供的示例代码
3. 使用评审清单记录发现的问题
4. 讨论评审结果
5. 分享评审经验和发现

### 5. 提交流程实践

#### 5.1 提交规范
```bash
# 格式：<type>(<scope>): <subject>
git commit -m "feat(core): add user authentication API"

# 类型说明：
# feat: 新功能
# fix: Bug修复
# docs: 文档更新
# style: 代码格式
# refactor: 代码重构
# test: 测试相关
# chore: 构建工具
```

#### 5.2 创建Pull Request
1. **标题**：清晰描述变更内容
2. **描述**：
   - 变更目的和背景
   - 测试情况
   - 相关任务链接
   - 截图或示例（如适用）
3. **标签**：添加适当标签
4. **审查者**：@mention至少2位审查者

#### 5.3 练习：完整提交流程
**任务**：完成一个完整的开发周期
1. 在worktree中开发一个小功能
2. 运行所有质量检查
3. 提交代码并推送到远程
4. 创建Pull Request
5. 请求同伴评审

### 6. 常见问题解决

#### 6.1 Git冲突解决
```bash
# 更新主分支
git checkout main
git pull origin main

# 解决冲突
git checkout feature-branch
git rebase main
# 解决冲突文件
git add .
git rebase --continue
```

#### 6.2 构建失败处理
1. **查看日志**：`cat build.log`
2. **常见原因**：
   - 依赖版本冲突
   - TypeScript类型错误
   - 测试失败
3. **解决方法**：
   - 更新依赖版本
   - 修复类型定义
   - 修复测试代码

#### 6.3 紧急修复流程
1. 从main创建hotfix分支
2. 最小化修改，重点测试
3. 快速评审和合并
4. 部署到生产环境
5. 后续补充完整测试

### 7. 团队协作最佳实践

#### 7.1 沟通规范
- **及时更新**：定期更新任务状态
- **明确责任**：明确任务负责人
- **寻求帮助**：遇到问题及时求助
- **分享知识**：分享经验和学习

#### 7.2 代码协作
- **小步提交**：频繁提交，每次解决一个问题
- **清晰注释**：代码和提交信息都要清晰
- **主动评审**：积极参与他人代码评审
- **接受反馈**：积极接受并回应评审反馈

#### 7.3 流程改进
- **定期回顾**：每月回顾流程效果
- **收集反馈**：收集团队成员的反馈
- **持续优化**：基于反馈优化流程
